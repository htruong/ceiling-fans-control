A bridge that exposes Casa Vieja ceiling fans (remote model `TR301A`) to
Home Assistant via an ESP32 + CC1101 RF radio. The remote has a big SET
button but no DIP switches, so we pair by sniffing the remote's unique
14-bit `FAN_ID` off the air and replaying it.

Two implementations live in this repo. Pick one:

1. **ESPHome + CC1101 + HA** (recommended). One ESP32-based RF node;
   all per-fan logic lives in Home Assistant; no firmware reflash to
   experiment. See below.
2. **Rust daemon on a Pi (legacy)**. Single Pi running `rpitx`/`sendook`
   on a GPIO pin; daemon bridges HA + native HomeKit. See [Legacy: Rust
   daemon on a Pi](#legacy-rust-daemon-on-a-pi) at the bottom.

Architecture (ESPHome path)
--

```
  Physical          304.30 MHz OOK         WiFi              Home
  TR301A remote ───────────────────►  ┌───────────┐ ────►  Assistant
                                      │  ESP32 +  │          │
  Ceiling fans  ◄───────────────────  │  CC1101   │ ◄────  REST API
                                      └───────────┘
                                      fan-remote.yaml
                                      (deployed once)
```

The ESP32 firmware (`fan-remote.yaml`) is a **dumb RF transceiver**:

- **RX**: pre-built `binary_sensor` patterns per (fan, command) so HA
  sees a state change when you press a physical remote. Patterns are
  generated by `utils/generate_esphome_config.py` from `config.yaml`.
- **TX**: one custom API action, `transmit_fan_bits(bits, repeat)`,
  takes a 25-bit string `"1111" + fan_id(14) + cmdid(7)` and encodes
  it on the fly. HA scripts compute the bits and call this action —
  no per-fan or per-command knowledge in the firmware.

Adding a new fan or changing a command:

| Change                | Touch                        | Reflash? |
| --------------------- | ---------------------------- | -------- |
| Add a new TX command  | HA script / `utils/ha_send.sh` | No     |
| Add a fan to control  | HA script / `utils/ha_send.sh` | No     |
| Receive a new fan's remote presses | `config.yaml` + regenerate `fan-remote.yaml` | Yes (RX only) |

Hardware
--

ESP32 (esp32dev / DevKit-C) + CC1101 sub-GHz module:

| ESP32 GPIO | CC1101 pin | Note                                  |
| ---------- | ---------- | ------------------------------------- |
| GPIO18     | SCK        |                                       |
| GPIO23     | MOSI       |                                       |
| GPIO19     | MISO       |                                       |
| GPIO5      | CSN        | strapping pin — harmless warning      |
| GPIO22     | GDO0       | TX data into CC1101                   |
| GPIO21     | GDO2       | RX data out of CC1101                 |
| 3V3, GND   | VCC, GND   |                                       |

Antenna: a 24.6 cm wire on the ANT pad works fine at 304.30 MHz; a tuned
helical or whip is better if range is tight.

Flash the firmware
--

1. Copy `config_sample.yaml` to `utils/config.yaml` and fill in real
   14-bit FAN_IDs (sniff them with `utils/decode_fan_remote.py` or pair
   arbitrary IDs via the SET-button procedure below).
2. Regenerate the `binary_sensor:` block in `fan-remote.yaml`:
   ```
   python3 utils/generate_esphome_config.py > /tmp/sensors.yaml
   # Paste under `binary_sensor:` in fan-remote.yaml.
   ```
   Or for one fan only:
   ```
   python3 utils/generate_esphome_config.py --fan parents_room --no-header
   ```
3. Flash with `esphome run fan-remote.yaml`. ESPHome's built-in CC1101
   component (since [esphome/esphome#11849](https://github.com/esphome/esphome/pull/11849))
   is used — no `external_components` needed.
4. Verify in Home Assistant → Settings → Devices → ESPHome that
   `fan_remote1` is online and the binary_sensors appear.

Sending a fan command
--

From a shell:

```
utils/ha_send.sh parents_room light
utils/ha_send.sh living_room speed3 6       # repeat 6 times
HA_URL=http://10.0.0.5:8123 utils/ha_send.sh upstairs stop
```

From an HA automation:

```yaml
service: esphome.fan_remote1_transmit_fan_bits
data:
  bits: "1111010111111010000000010"   # 4 preamble + 14 fan_id + 7 cmdid
  repeat: 4
```

Receiving a remote press
--

Each `binary_sensor` in `fan-remote.yaml` corresponds to one
(fan, button) pair. When the matching pattern is heard on air, the
sensor's state goes from `off` to `on` for ~500ms (debounced with
`delayed_off: 500ms` so the remote's 3-5 retransmits per press only
fire one HA state change). Build HA automations against
`binary_sensor.fan_<room>_<command>` state changes.

Pairing a fan with an arbitrary ID
--

Per the remote's manual, the fan learns whichever 14-bit `FAN_ID` the
remote transmits during a pairing window:

1. Remove the battery cover on the remote — there's a "Learn Switch"
   underneath.
2. Cut power to the ceiling fan at the breaker (or wall switch) for
   one minute.
3. Restore power. You have 60 seconds to pair.
4. Press and hold the Learn Switch with a pen or paperclip until the
   light on the fan blinks twice, then release.

You can also send the `pair` command (`0100100`) from this bridge
with your chosen 14-bit FAN_ID during the pairing window — the fan
should latch onto it. I've never actually tried this; I sniffed my
existing remotes instead. If you try it, please open an issue.

Decoding an existing remote's ID
--

If you'd rather mirror an existing remote than assign a new ID, either:

- Capture with an RTL-SDR dongle (follow
  [this tutorial](https://www.youtube.com/watch?v=_GCpqory3kc)) and
  decode with `utils/decode_fan_remote.py`, or
- Flash `fan-remote.yaml` with a placeholder FAN_ID, watch the
  ESPHome logs at DEBUG level, press the unknown remote, and read
  the 25-bit decoded message from the `Received Raw:` lines.

Protocol
--

304.30 MHz OOK, 333 µs per chip, 25-bit messages = preamble(4) +
FAN_ID(14) + cmdid(7). Each logical bit `b` encodes as 3 chips: `'1',
'0', b`. Full details in [protocol.txt](protocol.txt).

Debugging
--

The fan didn't respond:

- Check ESPHome logs at DEBUG level for `Sending raw...` and confirm the
  CC1101 mode-switch lines around it (`set_idle`, `begin_tx`,
  `begin_rx`). The `on_transmit` / `on_complete` hooks in `fan-remote.yaml`
  handle the TX↔RX mode dance.
- Compare your transmitted pulse list to a captured remote press at the
  same fan. Lengths and durations should match within ~20%.
- If you suspect the fan lost its pairing (e.g. after a long power
  outage), re-pair it with the SET-button procedure above.

I see `Received Raw:` lines but no `Fan X Y pressed`:

- The patterns aren't matching. Bump `tolerance:` on the
  `remote_receiver` or compare the captured pulses to the expected
  pattern from `generate_esphome_config.py`.

I press a remote button once but HA toggles three times:

- The `delayed_off: 500ms` filter on each binary_sensor exists
  precisely to prevent this — confirm it's present in
  `fan-remote.yaml`. The remote sends each press 3-5 times.

Legacy: Rust daemon on a Pi
--

The original implementation runs as a Rust daemon on a Raspberry Pi
(any model with GPIO; I used a Pi 0) and uses
[rpitx](https://github.com/F5OEO/rpitx)'s `sendook` to transmit OOK
straight off a GPIO pin
([no external hardware needed](https://www.youtube.com/watch?v=3lGU7PjJM7k)).

```
HA WebSocket  ─┐
               ├─►  onlyfansd (Rust)  ─►  sendook  ─►  Pi GPIO
HomeKit (HAP) ─┘    src/main.rs                       (304.30 MHz)
```

Build & deploy:

```
cargo zigbuild --release --target arm-unknown-linux-gnueabihf
# copy target/.../onlyfansd to /usr/local/bin/, config to /etc/onlyfansd/config.yaml,
# run under systemd.
```

The daemon also exposes a native HomeKit bridge (via vendored `hap-rs`)
for clients that prefer HAP over HA. Set `homeassistant.enabled: false`
to run HomeKit-only. See `src/`, `config_sample.yaml`, and
`ceiling_fans.yaml`.

This path predates the new RF/CC1101 support in HA + ESPHome and is
kept for setups that already work.
